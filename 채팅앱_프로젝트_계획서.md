## 1. 프로젝트 개요

| 항목 | 내용 |
|------|------|
| 프로젝트명 | (브랜드명은 본인 이름으로 — 예: `MyTalk`, `Chatto`) |

## 2. 목표와 범위

### 만들 것 (In Scope)
- 이메일 / 구글 로그인·회원가입
- 프로필 (닉네임, 프로필 사진, 상태메시지)
- 친구/유저 목록 및 검색
- 1:1 실시간 채팅
- 그룹 채팅
- 채팅방 목록 (마지막 메시지 미리보기, 안 읽은 메시지 수)
- 읽음 표시 (카톡의 "1")
- 이미지 전송
- 반응형 + 카톡 스타일 UI (말풍선, 노란 테마)
- 무료 호스팅 배포

### 안 만들 것 (Out of Scope) — 1차 범위에서 제외
- 음성 / 영상 통화 (WebRTC, 1인·무료 개발 난이도 급상승)
- 결제 / 선물하기
- 오픈채팅, 채널, 이모티콘 스토어
- 푸시 알림 (여유 되면 후순위로 추가 검토)

> 범위를 명확히 끊는 것이 1인 프로젝트 완주의 핵심. 통화 기능은 "향후 확장"으로 문서에만 남긴다.

## 4. 시스템 구성

```
[브라우저: React 앱 (Vercel 호스팅)]
        │  @supabase/supabase-js (anon 키)
        ▼
[Supabase]
  ├─ Auth        : 이메일 / 구글 로그인
  ├─ Postgres DB : profiles / chats / chat_members / messages
  ├─ Realtime    : 메시지 INSERT 실시간 푸시
  ├─ Storage     : 이미지 파일
  └─ RLS         : 행 단위 접근 권한 (보안 핵심)
```

서버를 직접 구축하지 않고 Supabase가 백엔드 역할을 전부 담당한다. React 앱은 `anon` 키로 Supabase에 직접 접근하며, 권한 통제는 RLS가 책임진다.

## 5. 데이터 모델

총 4개 테이블.

### profiles
Supabase Auth의 `auth.users`는 직접 수정 불가 → 부가 정보를 담을 테이블을 1:1로 연결. 회원가입 시 트리거로 자동 생성.

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid (PK, FK→auth.users) | 유저 식별자 |
| username | text | 닉네임 |
| avatar_url | text | 프로필 사진 경로 |
| status_message | text | 상태메시지 |
| created_at | timestamp | |

### chats
채팅방. 1:1과 그룹을 `type`으로 구분.

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid (PK) | 방 식별자 |
| type | text | `direct` / `group` |
| name | text | 그룹방 이름 (1:1은 null) |
| created_at | timestamp | |

### chat_members (★ 핵심)
유저↔채팅방은 다대다 → 연결 테이블. "이 방에 누가 있는가"를 표현. 그룹 채팅이 자연스럽게 해결됨.

| 컬럼 | 타입 | 설명 |
|------|------|------|
| chat_id | uuid (FK→chats) | |
| user_id | uuid (FK→profiles) | |
| last_read_at | timestamp | 어디까지 읽었는지 → "안 읽은 수" 계산의 근거 |
| joined_at | timestamp | |

### messages
| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid (PK) | |
| chat_id | uuid (FK→chats) | 어느 방 |
| sender_id | uuid (FK→profiles) | 누가 보냈는지 |
| content | text | 메시지 내용 |
| type | text | `text` / `image` |
| created_at | timestamp | 정렬 기준 |

**안 읽은 메시지 / 읽음("1") 로직:** 각 멤버의 `last_read_at` 이후에 생성된 메시지 수 = 안 읽은 개수. 상대가 방을 열어 `last_read_at`이 갱신되면 "1"이 사라지는 방식.

## 6. 핵심 기능 명세

| 기능 | 설명 | 사용 Supabase 기능 |
|------|------|--------------------|
| 회원가입/로그인 | 이메일·구글 | Auth |
| 프로필 설정 | 닉네임/사진/상태메시지 수정 | DB + Storage |
| 유저 검색 | 닉네임으로 검색해 대화 시작 | DB |
| 1:1 채팅 | 메시지 송수신 | DB + Realtime |
| 그룹 채팅 | 여러 명이 한 방에서 대화 | DB + Realtime |
| 채팅방 목록 | 마지막 메시지 + 안 읽은 수 | DB |
| 읽음 표시 | `last_read_at` 기반 "1" | DB |
| 이미지 전송 | 이미지 업로드 후 메시지로 전송 | Storage + DB |
| 실시간 수신 | 새로고침 없이 메시지 도착 | Realtime |

## 7. 개발 로드맵 (단계별 마일스톤)

각 단계마다 "보이는 결과물"이 남도록 구성 → 포트폴리오 캡처 누적.

- **0단계 — 셋업**
  Vite로 React 프로젝트 생성, Supabase 프로젝트 개설, `@supabase/supabase-js` 연결, 환경변수(`anon` 키) 설정, GitHub 저장소 + Vercel 배포 파이프라인 1회 구성.

- **1단계 — 정적 UI 클론** *(프론트 경험으로 바로 가능)*
  채팅방 목록 화면, 채팅방 화면(말풍선: 내 것은 오른쪽 노란색 / 상대는 왼쪽 흰색), 친구 탭, 하단 네비게이션. 더미 데이터로.

- **2단계 — 인증**
  이메일 회원가입/로그인 → 구글 OAuth 추가. 로그인 상태에 따른 라우팅 보호.

- **3단계 — 1:1 실시간 메시지 (★ 핵심 마일스톤)**
  `messages` 저장 + Realtime 구독으로 실시간 수신. 여기까지면 "되는 앱".

- **4단계 — 채팅방 목록 고도화**
  마지막 메시지 미리보기, 안 읽은 메시지 수 뱃지.

- **5단계 — 프로필 + 이미지**
  프로필 편집, Storage로 프로필 사진 / 이미지 메시지 전송.

- **6단계 — 그룹 채팅**
  `chat_members`를 활용한 다인원 방 생성·참여.

- **7단계 — 마무리 & 배포**
  읽음 표시("1"), 메시지 검색, 이모지, UI 다듬기, 최종 배포 점검.

- **(확장) 푸시 알림** — 여유 시 검토.

## 8. 보안 체크리스트 (★ 반드시)

Supabase는 DB가 클라이언트에 직접 노출되므로 보안 설정이 필수.

- [ ] **모든 테이블에 RLS(Row Level Security) 활성화**
- [ ] "방 멤버만 해당 방의 메시지 SELECT 가능" 정책 작성
- [ ] "본인 프로필만 UPDATE 가능" 정책 작성
- [ ] "방 멤버만 메시지 INSERT 가능" 정책 작성
- [ ] 프론트엔드에는 **`anon` 키만** 사용 (절대 `service_role` 키 노출 금지)
- [ ] `.env` 파일을 `.gitignore`에 포함 → 키가 GitHub에 올라가지 않게
- [ ] Storage 버킷 접근 권한 설정 (공개/비공개 구분)

> RLS로 권한을 분리했다는 점은 포트폴리오에서 좋은 어필 포인트가 된다.
